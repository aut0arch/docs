# aut0arch Benchmarks

This page documents the baseline performance of the `aut0arch` core modules (Parser, Clustering, and Explainer). We benchmarked execution time scaled against different repository sizes and codebase complexity.

## Parser Performance

The parsing step uses `tree-sitter` to read all `.java` files, extract syntax trees, and build the initial dependency graph consisting of nodes (classes, interfaces, methods) and edges (calls, containment). 

| Repository | Size Category | Files | Lines of Code (LOC) | Nodes Generated | Edges Generated | Parser Time (s) |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| `JacMattisen/board-java` | Small | 27 | 1,168 | 95 | 207 | 0.19 |
| `SaurabhDalavi16/Java-Project` | Small | 8 | 1,455 | 97 | 204 | 0.12 |
| `TheAlgorithms/Java` | Large | 1,548 | 134,034 | 10,760 | 145,643 | 78.31 |

**Parser Takeaway:** The time required to parse a project heavily scales with its size (Files & LOC). Massive repositories like `TheAlgorithms/Java` can take over a minute due to the deep semantic processing and link resolution required across over a hundred thousand lines of code.

## Clustering Performance

The clustering step ingests the dense graph generated by the parser and applies the Louvain modularity algorithm to group the raw nodes into cohesive architectural clusters.

| Repository | Nodes Processed | Edges Processed | Clusters Formed | Clustering Time (s) |
| :--- | :--- | :--- | :--- | :--- |
| `JacMattisen/board-java` | 95 | 207 | 15 | 0.19 |
| `SaurabhDalavi16/Java-Project` | 97 | 204 | 3 | 0.18 |
| `TheAlgorithms/Java` | 10,760 | 145,643 | 180 | 0.75 |

**Clustering Takeaway:** The modularity clustering algorithm is extremely fast and efficient. It scales remarkably well, taking under a second to process a graph with over 10,000 nodes and 145,000 edges!

## Explainer Performance

The explainer step passes the clustered code to a Large Language Model (LLM) to intelligently infer architectural responsibilities, inputs, outputs, and generate an overarching system story. We benchmarked the available OpenRouter APIs using the `JacMattisen/board-java` repository (which generates 15 clusters).

| Model | Time (s) | Status |
| :--- | :--- | :--- |
| `openrouter-openai/gpt-oss-20b:free` | 122.93 | success |
| `openrouter-stepfun/step-3.5-flash:free` | 31.84 | success |

**Explainer Takeaway:** The explainer module is heavily bottlenecked by API latency and rate limits. Lighter models like `step-3.5-flash` perform significantly faster (under 32s) compared to heavier models like `gpt-oss-20b` (over 2 minutes). Concurrency (`--workers`) speeds this up, but free-tier APIs often require artificial delays to avoid rate-limiting.
